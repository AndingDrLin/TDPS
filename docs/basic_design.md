# TDPS 总体设计

本文档描述 TDPS 的系统设计和工程约束。执行命令与回归流程不在本页维护，见 `docs/testing/`。

## 1. 项目目标

系统目标是在统一控制框架下完成三类任务：

1. 巡线行驶
2. 指定窗口 LoRa 发送
3. 障碍区前雷达决策与绕行

当前仓库已实现并可回归验证：

- 任务 1：巡线（8 路灰度）
- 任务 2：LoRa 检查点发送（异步队列 + 超时重试 + 可选 ACK）
- 任务 3：雷达串口帧解析与避障状态输出（保守默认协议）

## 2. 架构决策

### 2.1 软件组织

采用“主循环 + 外设中断 + 状态机”的结构，不引入 RTOS。

原因：

- 任务规模可控，状态边界明确
- 调试成本低
- 便于在课程节奏下快速迭代

### 2.2 分层

- 平台层：外设访问与板级适配
- 算法层：传感器处理、PID 控制、底盘输出
- 应用层：状态机与异常处理
- 扩展层：LoRa/雷达事件钩子 + 非阻塞服务 Tick

### 2.3 可测试性优先

核心控制逻辑必须能脱离实机在离线 harness 中回归，避免“上板才能验证”。

## 3. 硬件假设

- 差速底盘（左右独立驱动）
- 8 路灰度巡线传感器（默认 `LF_SENSOR_COUNT=8`）
- STM32 主控
- LoRa 模块（EWM22A-900BWL22S）
- 24GHz 雷达模块（HLK-LD2410S）

## 4. 巡线主链路

巡线流程：

1. 上电后等待启动条件
2. 进入标定窗口，左右摆动采样
3. 标定完成后进入 `RUNNING`
4. `RUNNING` 中按偏差 PID 输出差速
5. 丢线进入 `RECOVERING`
6. 恢复成功回到 `RUNNING`；超时进入 `STOPPED`

该流程在 `line_follow_v1` 中已实现，并由离线测试覆盖。

## 5. LoRa / 雷达接入策略

不直接在巡线控制链路里插入业务逻辑；统一走钩子接口：

- `LF_Hook_OnReservedCheckpoint`：通信窗口
- `LF_Hook_OnReservedObstacleWindow`：障碍决策窗口

原则：

- 扩展逻辑不得破坏巡线主循环周期
- 扩展逻辑不得直接绕过底盘控制接口

## 6. 验证策略

### 6.1 离线门禁

- quick：单次门禁
- stability：多 seed 最差值门禁
- baseline：版本间退化保护

### 6.2 实机验证

- 先验证外设与运动方向
- 再验证标定与巡线闭环
- 最后验证任务切换和扩展逻辑

## 7. 里程碑

1. M1：底盘与外设联通
2. M2：基础巡线闭环
3. M3：丢线恢复稳定
4. M4：LoRa 窗口发送接入
5. M5：雷达决策接入
6. M6：三任务统一联调
